1. Descripción General

El Sistema de Gestión de Cupos de Viajes Aéreos es una aplicación web segura desarrollada con React y Supabase, diseñada para la gestión de cupos de viajes aéreos corporativos. La aplicación permite a los usuarios solicitar reservas de viajes y a los administradores gestionar usuarios y disponibilidad, manteniendo integración con Power Automate para el procesamiento de solicitudes.

Estado Actual: ✅ Producción Ready - Todas las vulnerabilidades críticas han sido corregidas y el sistema está listo para deployment empresarial.

2. Características Principales

Funcionalidades del Sistema

  • Gestión de Usuarios: Autenticación segura con Supabase Auth, creación de usuarios (solo administradores), gestión de perfiles con roles y agencias, autorización basada en roles
  • Gestión de Reservas: Consulta de disponibilidad en tiempo real, solicitud de reservas con validación, seguimiento de confirmaciones, filtrado por agencia automático
  • Seguridad y Administración: Panel de administración protegido, rutas privadas con verificación de roles, logs de auditoría en Edge Functions, validación de datos robusta
  • Conexiones Dinámicas: Soporte para múltiples APIs (Power Automate, Supabase, MongoDB, etc.), encriptación zero-knowledge de credenciales, sistema de conexión activa por usuario
  • Mapeo de Datos: Conversión automática entre formatos de API, validación de estructura estándar, compatibilidad con formato SharePoint actual

4. Sistema de Seguridad

Resumen del Sistema de Seguridad

El sistema implementa un sistema de seguridad empresarial que incluye:

  • Sistema de bloqueo por intentos fallidos: Protección contra ataques de fuerza bruta
  • Auto-logout por inactividad: Gestión automática de sesiones inactivas
  • Autenticación de doble factor (2FA): Capa adicional de seguridad con TOTP
  • Panel administrativo: Herramientas de gestión y monitoreo de seguridad

Base de Datos de Seguridad

Tablas de Seguridad

-- Tabla para registrar intentos de login
user_login_attempts (
  id, user_id, ip_address, user_agent,
  success, failure_reason, created_at
)

-- Tabla para estado de seguridad de usuarios
user_security_status (
  user_id, is_locked, locked_at, failed_attempts,
  last_login_attempt, unlock_at, created_at, updated_at
)

-- Tabla para gestión de sesiones
user_sessions (
  id, user_id, session_token, last_activity,
  ip_address, user_agent, expires_at,
  created_at, updated_at
)

Funciones SQL Principales

  • log_login_attempt(): Registra cada intento de login
  • check_and_update_lock_status(): Gestiona el estado de bloqueo
  • unlock_user(): Desbloquea usuarios manualmente
  • cleanup_expired_sessions(): Limpia sesiones expiradas

Flujos de Trabajo de Seguridad

Flujo de Login Seguro

  1. Usuario ingresa credenciales
  2. Verificación de estado de bloqueo
    • Si está bloqueado → Mostrar mensaje de error
    • Si no está bloqueado → Continuar
  3. Validación de credenciales
    • Si falla → Registrar intento + incrementar contador
    • Si es exitoso → Verificar 2FA
  4. Verificación 2FA (si está habilitado)
    • Solicitar código TOTP o backup
    • Verificar código
  5. Crear sesión → Redirect a dashboard

Flujo de Auto-logout

  1. Monitoreo de actividad cada 30 segundos
  2. Verificación de última actividad
  3. Si > 10 minutos inactivo:
    • Mostrar advertencia (2 minutos antes)
    • Auto-logout y limpiar sesión
    • Redirect a login

Flujo de Configuración 2FA

  1. Usuario accede a configuración
  2. Generar secreto TOTP
  3. Mostrar código QR
  4. Usuario escanea con app autenticadora
  5. Verificar código de configuración
  6. Generar códigos de backup
  7. Habilitar 2FA en cuenta

5. Sistema de Roles y Permisos

Roles Definidos

Admin (admin)

Descripción: Administrador del Sistema

Permisos: Acceso completo a todas las funcionalidades

Accesos:

  • Ve todos los datos de todas las agencias
  • Gestión de usuarios
  • Gestión de conexiones API
  • Disponibilidad, solicitudes y confirmaciones globales
  • Configuración del sistema

Agency Admin (agency_admin)

Descripción: Administrador de Agencia

Permisos: Gestión de datos de su agencia

Accesos:

  • Ve datos solo de su agencia
  • Disponibilidad global (todos los vuelos)
  • Solicitudes de su agencia
  • Confirmaciones de su agencia

Restricciones:

  • No puede gestionar usuarios
  • No puede gestionar conexiones API

Agency User (agency_user)

Descripción: Usuario de Agencia

Permisos: Acceso limitado a sus propios datos

Accesos:

  • Disponibilidad global (todos los vuelos)
  • Solo sus propias solicitudes

Restricciones:

  • No puede ver confirmaciones
  • No puede gestionar usuarios
  • No puede gestionar conexiones API

6. Sistema de Conexiones Dinámicas

Descripción del Sistema

El sistema implementa una arquitectura de conexiones dinámicas que permite a los usuarios conectar diferentes fuentes de datos sin depender de variables de entorno hardcodeadas. Esto resuelve la contradicción entre un sistema con endpoints fijos y la necesidad de flexibilidad para conexiones personalizadas por usuario.

Evolución del Sistema

Fase 1: Sistema Actual (Variables Hardcodeadas)

Frontend → Variables .env → Power Automate endpoints fijos

Fase 2: Sistema Híbrido (Implementado Ahora)

Frontend → Conexión Activa? → Endpoints dinámicos
         ↓ (si no existe)
         → Variables .env → Endpoints fijos (fallback)

Fase 3: Sistema Completamente Dinámico (Futuro)

Frontend → Conexión Activa → Cualquier API (PA, Supabase, MongoDB, etc.)
         ↓ (sin fallback)
         → Error controlado si no hay conexión

Beneficios de la Nueva Arquitectura

1. Flexibilidad Total

// Un usuario puede usar Power Automate:
const conexionPA = {
  type: "powerautomate",
  credentials: {
    pedidosUrl: "https://prod-xx.powerautomate.com/flows/pedidos",
    productosUrl: "https://prod-xx.powerautomate.com/flows/productos",
  },
};

// Otro usuario puede usar Supabase:
const conexionSupabase = {
  type: "supabase",
  credentials: {
    url: "https://xyz.supabase.co",
    apiKey: "eyJhbGciOiJ...",
  },
};

2. Independencia de Entorno

  • Desarrollo: Variables .env como fallback
  • Producción: Conexiones dinámicas por usuario
  • Testing: Conexiones mockeadas

3. Escalabilidad

// Soporte futuro para múltiples fuentes simultáneas:
const usuario1 = { conexionActiva: "powerautomate" };
const usuario2 = { conexionActiva: "supabase" };
const usuario3 = { conexionActiva: "mongodb" };

10. API y Endpoints

Backend Flexible API

Endpoints de Datos

GET /api/data

Obtiene datos de una tabla específica con posibilidad de filtrado, límites y ordenamiento.

  • Parámetros Query:
    • table: Nombre de la tabla (requerido)
    • filters: Objeto JSON con filtros
    • limit: Número máximo de registros
    • offset: Desplazamiento para paginación
    • order: Campo y dirección de ordenamiento (ej: "created_at:desc")
POST /api/data

Realiza operaciones de inserción o actualización de datos.

{
  "table": "nombre_tabla",
  "operation": "insert", // o "update"
  "data": {
    "campo1": "valor1",
    "campo2": "valor2"
  },
  "id": "id_registro", // para actualización
  "idField": "id" // campo a usar como identificador
}
PUT /api/data

Realiza operaciones de actualización de datos (mismo formato que POST).

DELETE /api/data

Elimina un registro específico.

{
  "table": "nombre_tabla",
  "operation": "delete",
  "id": "id_registro",
  "idField": "id" // campo a usar como identificador
}
POST /api/data/query

Ejecuta consultas SQL personalizadas (solo lectura).

{
  "query": "SELECT * FROM productos WHERE categoria = $1",
  "params": ["viajes"]
}
GET /api/data/schema/{table}

Obtiene el esquema de una tabla específica.

GET /api/data/tables

Obtiene la lista de todas las tablas disponibles en la base de datos.

Filtros Soportados

Los filtros se pasan como objeto JSON y admiten los siguientes operadores:

  • $eq: Igualdad ({ "estado": { "$eq": "activo" } })
  • $ne, $neq: Diferente ({ "estado": { "$ne": "inactivo" } })
  • $gt: Mayor que ({ "precio": { "$gt": 100 } })
  • $gte: Mayor o igual que ({ "precio": { "$gte": 100 } })
  • $lt: Menor que ({ "precio": { "$lt": 500 } })
  • $lte: Menor o igual que ({ "precio": { "$lte": 500 } })
  • $in: Incluido en una lista ({ "categoria": { "$in": ["viajes", "hoteles"] } })
  • $like: Búsqueda parcial ({ "titulo": { "$like": "vacaciones" } })
  • $ilike: Búsqueda parcial insensible a mayúsculas ({ "titulo": { "$ilike": "VACACIONES" } })

11. Flujos de Trabajo

Flujo de Operación de Usuario

Usuario Admin → Frontend Service → Edge Function → Supabase DB
     ↑                                ↓
   Respuesta ← UI Update ← Response ← Validation

Flujo de Operación de Reserva

Usuario → Reservation Service → Power Automate Proxy → Power Automate
    ↑                              ↓
  Respuesta ← UI Update ← Filtered Response ← External API

Flujo de Datos con Conexiones Dinámicas

Prioridad de Selección de Endpoints

  1. Conexión activa (dinámico) - Futuro
  2. Variables de entorno (fallback) - Presente
  3. Error si no hay ninguna - Fallo controlado

Diferentes Endpoints por Tipo de Datos

// Antes (problemático):
getDataFromActiveConnection(); // Mezclaba todos los datos

// Ahora (correcto):
getDataFromActiveConnection("pedidos"); // Tabla pedidos (solicitudes/confirmaciones)
getDataFromActiveConnection("productos"); // Tabla productos (disponibilidad)

  • Fuerza Bruta: Bloqueo automático después de 3 intentos fallidos
  • Session Hijacking: Tokens únicos por sesión + validación IP
  • CSRF: Tokens de sesión seguros
  • Replay Attacks: Códigos TOTP de un solo uso

Monitoreo y Auditoría

  • Registro completo de intentos de login
  • Tracking de IP y User Agent
  • Estadísticas de seguridad en tiempo real
  • Alertas de actividad sospechosa

Gestión de Sesiones

  • Limpieza automática de sesiones expiradas
  • Múltiples sesiones por usuario (opcional)
